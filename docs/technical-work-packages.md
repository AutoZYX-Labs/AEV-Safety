# AEV Safety 技术工作包与证据追踪

[English](technical-work-packages-en.md)

## 目的

本文件把 AEV Safety（自主工程车辆安全）的公开研究框架拆成可执行、可审查、可复用的技术工作包。它的作用不是把开放问题收束为单一论文，而是让研究人员、测试评价机构、标准化专家和企业工程团队能够沿着同一条证据链继续推进。

工作包遵循一个基本原则：

每一个公开 claim 都要能够追踪到输入材料、技术对象、验证方式、证据边界和后续工作。

## WP0：证据治理与公开边界

目标：防止把内部材料、公开演示、标准状态页、企业官网和同行评议论文混成同一等级证据。

核心问题：

1. 哪些来源可以支撑公开事实？
2. 哪些来源只能作为背景输入？
3. 哪些结论必须等待授权标准全文、正式测试报告或现场运行数据？

当前输出：

- 四级证据分类：A-FULL、B-OFFICIAL、C-PUBLIC、D-INTERNAL。
- 公开网页和论文中的证据边界说明。
- 对未公开商业材料的保守使用规则。

质量门：

- 每个标准状态、项目状态和日期必须可追踪到官方页面或授权文本。
- 内部材料不得被写成安全性能、认证成熟度或市场结论。
- 新闻、视频和公开演示只能作为场景锚点，不能作为安全有效性证据。

## WP1：MWMS 系统边界定义

目标：把安全对象从单台车辆扩展为移动作业机器系统。

MWMS 包括：

1. 车辆平台：动力、制动、转向、液压、底盘、上装和控制器。
2. 工作装置：铲斗、动臂、斗杆、打桩装置、开沟装置、压实装置、破碎装置等。
3. 工具与物料：料堆、土石方、矿石、车厢、料斗、桩基、沟槽、太阳能支架等。
4. 场地基础设施：围挡、地理围栏、定位、通信、调度、地图、能源补给和警示设施。
5. 人员与组织：现场人员、维护人员、远程监督员、远程驾驶员、调度员和第三方协同人员。
6. 外部系统：卡车、输送带、无人机、机器狗、站点服务器、矿山/工地管理系统。

当前输出：

- `architecture/reference-architecture.md` 的五层结构。
- `paper/manuscript-zh.md` Section 5.1。
- README 中的 MWMS 定义。

质量门：

- 任何测试记录都必须说明 MWMS 边界，而不是只写机器型号。
- 任何安全 claim 都必须说明它覆盖的是整机、工作装置、场地系统还是组织流程。

## WP2：WTDC 与 DMWT 分类体系

目标：把道路车辆 ODD/DDT 方法迁移到作业场地与任务。

WTDC 应覆盖：

1. 场地：矿山、港口、拌合站、施工现场、光伏场地、船舱、沙盘、月面类测试场。
2. 地形：坡度、路面、低附着、坑洼、边坡、沟槽、松散物料和未知地形。
3. 物料：砂石、矿石、土方、泥浆、粉尘、桩基、太阳能支架和未知星球表层材料。
4. 人员和资产：工人、旁观者、人工驾驶设备、卡车、输送带、固定设施和禁入区。
5. 运行条件：通信、定位、地图新鲜度、能源、天气、照明、远程监督和调度约束。

DMWT 应覆盖：

1. 移动：前进、倒车、转弯、V 型循环、坡道、避障、跟车和队列。
2. 作业：铲掘、装载、卸料、运输、打桩、开沟、压实、清舱、破碎和材料转运。
3. 任务质量：满载率、卸料偏差、开沟深度、桩位、压实度、平整度和返工。
4. 恢复：减速、停止、急停、远程确认、远程驾驶、现场接管、撤离和复测。

当前输出：

- `taxonomy/worksite-task-taxonomy-v1.0.md`。
- `benchmarks/evaluation-matrix.md` 的测试记录字段。
- `paper/manuscript-zh.md` Section 5.2 和 5.3。

质量门：

- 场景不能只写为“装载机作业”，必须给出 WTDC 和 DMWT 的可复核字段。
- 测试通过准则必须连接任务后果，而不是只报告碰撞或急停。

## WP3：SOTIF 导向风险链

目标：把功能不足、触发条件、危险事件、任务后果和缓解措施连接起来。

典型风险链：

```text
WTDC/DMWT 条件
→ 功能不足
→ 触发条件
→ 危险事件或任务失败
→ 六类后果
→ 缓解措施
→ 剩余风险和证据
```

示例：

| 场景 | 功能不足 | 触发条件 | 后果 | 缓解 |
|---|---|---|---|---|
| 料堆铲装 | 料堆形态识别不足 | 粉尘、背光、物料坍塌 | 满载率过低、撒料、液压冲击 | 降速、重新估计料堆、远程确认 |
| 船舱清舱 | 人员/障碍物识别不足 | 狭窄空间、遮挡、强反光 | 人员暴露、资产碰撞、恢复困难 | 禁入区、低速模式、现场确认 |
| 光伏打桩 | 桩位和地形判断不足 | 地面不平、GPS 漂移、支架遮挡 | 桩位偏差、返工、设备损伤 | 复测定位、任务暂停、人工复核 |
| 月面资源作业 | 地形/物料未知 | 长延迟、低能源、通信中断 | 设备困陷、任务中断、无法恢复 | 预案路径、局部自主恢复、能源保护 |

当前输出：

- `paper/manuscript-zh.md` Section 5.4。
- `standards/standardization-roadmap.md` 中的 SOTIF 风险链条款骨架。
- `benchmarks/evaluation-matrix.md` 中的 trigger_condition 和 insufficiency 字段。

质量门：

- 不把 SOTIF 写成碰撞风险清单；必须包含功能不足和触发条件。
- 不把道路车辆 ODD 直接等同于 WTDC；必须覆盖物料、工具、任务质量和恢复能力。

## WP4：测试评价与运行监控

目标：让安全证据从一次性试验扩展到开发、封闭场地、现场运行和复盘闭环。

评价对象：

1. 人员安全。
2. 资产与基础设施。
3. 生产连续性。
4. 任务质量。
5. 环境与资源。
6. 恢复能力。

数据来源：

1. 仿真场景。
2. 封闭场地试验。
3. 室内/室外实车测试。
4. 现场运行日志。
5. 人工评审与远程监督记录。
6. 事故、近失事件和异常复盘。

当前输出：

- `benchmarks/evaluation-matrix.md`。
- `paper/manuscript-zh.md` Section 6.2、6.3 和 6.4。

质量门：

- 指标必须说明单位、采样频率、触发条件、适用机器和适用场地。
- 阈值不得在没有机器、场地、任务和授权标准依据时写成通用标准。
- 运行监控必须能够回填 taxonomy、风险链和 safety case。

## WP5：Safety Case 与标准化模块

目标：把研究框架转化为可被标准化专家、测试机构和企业工程团队审查的证据结构。

候选标准模块：

1. 范围与术语：MWMS、WTDC、DMWT、功能不足、触发条件、任务后果、最小风险状态。
2. 系统边界：机器、工具、物料、场地、人员、监督和外部协同系统。
3. 场景分类：功能场景、逻辑场景和具体场景在作业任务中的表达。
4. 风险分析：SOTIF 风险链、功能安全失效、网络安全事件和 AI 功能不足的边界。
5. 测试评价：仿真、封闭场地、现场运行、运行监控和近失事件复盘。
6. 安全论证：claims、arguments、evidence、assumptions、limitations 和 residual risk。

当前输出：

- `standards/landscape.md`。
- `standards/standardization-roadmap.md`。
- `paper/manuscript-zh.md` Section 6.5 和 9.3。

质量门：

- 所有标准名称、版本、项目状态和公示周期必须按官方来源核验。
- 不把相邻标准写成直接合规依据。
- 不把候选条款写成已经发布或已经被行业采纳的结论。

## WP6：工程落地与产品化证据

目标：让研究框架能服务真实产品，而不是只停留在论文表达。

面向的工程对象：

1. 装载机自主装卸。
2. 挖掘机和开沟设备。
3. 矿卡和自动运输系统。
4. 光伏施工与打桩设备。
5. 清舱、破碎、压实、材料转运机器人。
6. 极端环境和未来星球资源作业系统。

产品化证据应包括：

1. 需求分解：从 WTDC、DMWT 和后果类别导出安全需求。
2. 接口定义：底层控制器、液压/制动/转向接口、远程监督接口和调度接口。
3. 降级恢复：功能不足、触发条件、低置信度、通信退化和现场接管流程。
4. 现场记录：测试记录、运行日志、异常样本、近失事件和复测结果。
5. 责任边界：整机开发、前装、后装、远程运营、现场管理和第三方协同责任。

质量门：

- 产品落地必须给出具体机器、场地、任务和运行模式。
- 不使用通用安全口号替代可审查的证据字段。
- 企业内部数据公开前必须完成脱敏、权属确认和证据等级标注。

## 如何使用本文件

| 使用者 | 建议路径 |
|---|---|
| 研究人员 | 从 WP1-WP3 提炼论文问题，从 WP4-WP5 设计方法验证，从 WP6 找工程落点 |
| 测试评价机构 | 从 WP2-WP4 设计场景库、测试记录和运行监控字段 |
| 标准化专家 | 从 WP0、WP3 和 WP5 讨论术语、范围、条款骨架和证据边界 |
| 企业工程团队 | 从 WP1、WP2、WP4 和 WP6 建立产品 safety case 与现场问题闭环 |

## 当前不做的事情

1. 不发布内部商业计划书、客户信息、成本、报价或未公开合作细节。
2. 不替代授权标准全文、正式测试报告、企业现场安全评审或认证结论。
3. 不把道路车辆自动驾驶安全标准直接搬成工程车辆合规要求。
4. 不把机器人 SOTIF 在研项目写成已发布工程车辆标准。
5. 不把论文发表当作项目唯一目标；真实产品落地和标准化共识同样重要。
