# AEV Safety 标准化机会与建议

[English](standardization-opportunities-en.md)

信息状态：2026-06-05。

本文件基于 AEV Safety 已完成的标准版图、材料吸收和论文初稿，提出后续标准化建议。它不是标准立项申请书，也不是任何标准组织的正式意见。它的作用是把“自主工程车辆安全”转化为可讨论、可拆分、可试点的标准化工作包。

## 核心判断

AEV Safety 的标准化机会不在于宣称工程车辆没有标准。相反，工程机械、矿山装备、机器人、道路车辆自动驾驶和人工智能（Artificial Intelligence, AI）安全已经有大量标准与指南。

真正的机会在于：

把现有标准中的机器安全、控制系统功能安全、道路车辆预期功能安全（Safety of the Intended Functionality, SOTIF）、场景测试、远程控制、矿山自主系统、机器人安全和 AI 安全，连接成面向移动作业机器系统的场景证据链。

这条证据链应覆盖：

1. MWMS：移动作业机器系统（Mobile Working Machine System）。
2. WTDC：作业场地与任务设计条件（Worksite and Task Design Conditions）。
3. DMWT：动态移动与作业任务（Dynamic Moving and Working Task）。
4. 预期功能安全风险链：功能不足、触发条件、危险事件、任务后果和缓解措施。
5. 测试评价：仿真、封闭场地、现场运行、运行监控和近失事件复盘。
6. 安全案例：主张、论证、证据、假设、限制和剩余风险。

## 标准对照形成的缺口

| 标准/指南族 | 已有价值 | 对 AEV Safety 的缺口 |
|---|---|---|
| ISO 26262 | 道路车辆功能安全生命周期、汽车安全完整性等级（Automotive Safety Integrity Level, ASIL）、安全目标和硬件/软件开发流程 | 适用对象是道路车辆电气/电子（Electrical/Electronic, E/E）系统，不能直接覆盖工程车辆工作装置、物料交互和施工现场组织 |
| ISO 21448 | 预期功能安全、功能不足、触发条件和已知/未知不安全场景 | 需要从道路运行设计域（Operational Design Domain, ODD）/动态驾驶任务（Dynamic Driving Task, DDT）迁移到 WTDC/DMWT |
| ISO 34502 | 场景分类和场景驱动测试语言 | 需要增加物料、工具、任务质量、恢复能力和现场组织维度 |
| ISO/PAS 8800 | 道路车辆 AI 安全生命周期 | 可启发 AI 组件安全，但不能替代工程车辆现场验证和作业质量评价 |
| ISO/SAE 21434 | 道路车辆网络安全工程 | 需要连接远程监督、调度系统、现场通信、地图和运行日志 |
| ISO 17757 | 土方机械和矿山自主/半自主机器系统安全 | 可作为工程机械自主系统底座，但仍需与预期功能安全、场景测试和安全案例连接 |
| ISO 19014 | 土方机械控制系统安全相关部件功能安全 | 适合控制系统失效风险，不足以覆盖功能不足和任务后果 |
| ISO 15817 | 土方机械远程操作者控制系统 | 支撑远程控制和操作者职责，但需要扩展到远程监督、控制权切换和运行监控 |
| ISO 21815 | 土方机械碰撞警告和碰撞避免 | 支撑人员/资产接近风险，不覆盖物料交互、任务质量和生产连续性 |
| ISO 18497-1:2024 | 农业机械和拖拉机的部分自动、半自主和自主功能设计原则与词汇 | 是相邻非道路自动化机械线索，不是工程车辆合规标准 |
| 机器人预期功能安全项目 | 运行设计域、场景、触发条件、功能不足、风险评价和缓解措施正在进入机器人标准化语境 | 是移动物理 AI 标准化信号，不是已发布工程车辆标准 |
| GMG 自主矿山指南 | 自主矿山、运营准备、变更管理和安全案例语境 | 更偏矿山运营，需要转化到装载、开沟、打桩、清舱、光伏施工等作业任务 |

## 优先标准化建议

### 建议 1：自主工程车辆预期功能安全实施指南

建议定位：

面向自主工程车辆和移动作业机器系统的预期功能安全方法指南。

建议范围：

1. 术语：MWMS、WTDC、DMWT、功能不足、触发条件、任务后果、最小风险状态。
2. 场景：作业场地、物料、工具、人员、协同设备、远程监督和恢复条件。
3. 风险链：功能不足 → 触发条件 → 危险事件/任务失败 → 后果 → 缓解。
4. 证据：仿真、封闭场地、实车、运行监控、近失事件和安全案例。

为什么优先：

机器人预期功能安全国家标准项目已经显示预期功能安全方法正在从道路车辆向移动物理 AI 扩展。工程车辆具备更明确的产业场景和测试对象，适合形成更具体的行业指南或团体标准。

### 建议 2：自主工程车辆作业场地与任务设计条件

建议定位：

定义 WTDC，用于替代直接套用道路车辆运行设计域。

建议范围：

1. 场地类型：矿山、港口、拌合站、施工现场、光伏场地、船舱、沙盘、月面类测试场。
2. 地形与物料：坡度、低附着、粉尘、料堆、坑槽、边坡、未知物料。
3. 人员与资产：工人、旁观者、人工设备、卡车、输送带、固定设施和禁入区。
4. 运行约束：通信、定位、地图、能源、照明、天气、调度和监督。
5. 不允许运行条件：超出 WTDC、监督不可用、定位漂移、禁入区失效、恢复路径不可用。

为什么优先：

没有 WTDC，测试机构和企业很容易把“工程车辆自动化”写成泛泛的场地描述，无法形成可复测的场景库和安全证据。

### 建议 3：自主工程车辆动态移动与作业任务分类

建议定位：

定义 DMWT，用于扩展道路车辆动态驾驶任务。

建议范围：

1. 移动任务：前进、倒车、转弯、V 型循环、坡道、避障、跟车、队列。
2. 作业任务：铲掘、装载、卸料、运输、打桩、开沟、压实、清舱、破碎、材料转运。
3. 任务质量：满载率、卸料偏差、开沟深度、桩位、压实度、平整度。
4. 恢复任务：减速、停止、急停、远程确认、远程驾驶、现场接管、撤离、复测。

为什么优先：

工程车辆安全的独特性恰恰在“作业”。如果没有 DMWT，标准化容易退回到碰撞避免、急停和远程控制，难以覆盖产品真实价值。

### 建议 4：自主工程车辆测试评价与运行监控要求

建议定位：

定义测试记录、运行监控、近失事件和安全案例证据字段。

建议范围：

1. 六类后果：人员安全、资产与基础设施、生产连续性、任务质量、环境与资源、恢复能力。
2. 测试记录：machine_type、worksite_type、task_type、WTDC、DMWT、trigger_condition、insufficiency、consequence、mitigation、evidence。
3. 运行监控：低置信度、分布外输入、远程监督请求、人工接管、最小风险状态、任务质量偏差和近失事件。
4. 回填机制：运行异常应回填到分类体系、风险链和安全案例。

为什么优先：

这部分最接近测试评价机构和企业工程实践。它可以先做成团体标准、测试指南或公开模板。

### 建议 5：自主工程车辆安全案例指南

建议定位：

定义从开发、测试到运行的安全论证结构。

建议范围：

1. 主张：系统在定义的 WTDC/DMWT 内可接受地安全。
2. 论证：通过功能安全、预期功能安全、网络安全、AI 安全、机器安全、运行监控和组织流程共同论证。
3. 证据：标准符合性、仿真、封闭场地、实车、运行日志、近失事件、故障恢复和人工评审。
4. 假设：场地边界、监督能力、通信质量、维护状态和人员职责。
5. 限制：不覆盖超出 WTDC、未经验证任务、未授权改装和未定义协同系统。

为什么优先：

安全案例可以把论文、测试和标准化连接起来，也适合与企业产品安全评审和第三方评价结合。

## 推荐推进顺序

| 阶段 | 时间 | 目标 | 输出 |
|---|---|---|---|
| Phase 0 | 2026-06 至 2026-07 | 完成公开白皮书和专家讨论材料 | 标准化机会分析、术语表、WTDC/DMWT 初稿 |
| Phase 1 | 2026-07 至 2026-09 | 形成团体标准预研包 | 立项建议、范围、标准框架、主要条款骨架、样例场景 |
| Phase 2 | 2026-09 至 2026-12 | 选择一个最小可落地标准主题 | 优先建议为“自主工程车辆测试评价与运行监控要求”或“自主工程车辆预期功能安全实施指南” |
| Phase 3 | 2027-H1 | 企业样例和测试机构试用 | 示例测试记录、近失事件模板、安全案例样例 |
| Phase 4 | 2027-H2 以后 | 向更高层级标准转化 | 行业标准、国家标准预研、ISO/IEC/SAE/IEEE 工作组交流材料 |

## 最小可行标准化包

如果只能优先推进一个标准化成果，建议选择：

自主工程车辆测试评价与运行监控要求

原因：

1. 比“安全保证总则”更容易落地。
2. 比“预期功能安全实施指南”更容易与测试机构和企业试验体系结合。
3. 可以自然包含 WTDC、DMWT、预期功能安全风险链和安全案例字段。
4. 可以通过装载机、挖掘机、矿卡、光伏打桩和清舱机器人样例逐步扩展。

建议章节：

1. 范围。
2. 规范性引用文件。
3. 术语和定义。
4. 总体原则。
5. 作业场地与任务设计条件。
6. 动态移动与作业任务。
7. 场景分类与触发条件。
8. 测试评价指标。
9. 运行监控与近失事件记录。
10. 安全论证证据要求。
11. 附录 A：测试记录模板。
12. 附录 B：典型任务样例。
13. 附录 C：证据等级和公开边界。

## 需要避免的表述

1. 不说“工程车辆没有标准”。
2. 不说“道路车辆标准可以直接适用于工程车辆”。
3. 不把 ISO 18497-1:2024 写成工程车辆合规依据。
4. 不把机器人预期功能安全项目写成已发布工程车辆标准。
5. 不把企业公开页面或演示视频写成安全认证证据。
6. 不把阈值、比例或 KPI 写成行业通用要求，除非有授权标准、试验数据或专家共识支持。

## 与论文和网页的关系

标准化工作不是论文之后的附属产物，而是 AEV Safety 的核心输出之一。

建议形成三条互相支撑的成果链：

1. 中文综述论文：解释问题版图和方法迁移。
2. 英文方法论文：面向高水平期刊，提出 MWMS/WTDC/DMWT 和预期功能安全风险链方法。
3. 标准化预研包：把论文框架转化为术语、场景、测试、运行监控和安全案例条款骨架。
